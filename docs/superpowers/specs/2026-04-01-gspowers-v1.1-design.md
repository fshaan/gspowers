# gspowers v1.1 设计规格：知识复利闭环

## 概述

gspowers v1.0 的 10 步流程运行稳定，但缺乏跨项目知识传递机制。v1.1 通过三个精确改动实现知识复利闭环：

1. **compound 步骤**：ship 后沉淀经验到 `docs/solutions/`
2. **知识搜索**：brainstorming 前自动搜索历史经验作为参考
3. **init 简化**：existing 项目用自然语言描述即可自动生成 Update_Plan.md

形成闭环：`compound → docs/solutions/ → 知识搜索 → brainstorming`

## 约束条件

- 不替换 superpowers 核心链（brainstorming / writing-plans / subagent-dev）
- 不替换 gstack /review
- 不增加 /clear 边界（compound 在 finish 阶段内）
- 向后兼容 v1.0 state.json
- 导航员模式不变（唯一例外：知识搜索的只读预处理）

---

## 改动 1：compound 步骤

### 步骤映射变更

```
ship → compound                                             [finish]
compound → document-release                                 [finish]
```

v1.0 的 `ship → document-release` 替换为 `ship → compound → document-release`。

### 流程地图变更

```
★ 收尾期 ──────────── /clear ────
├─ [{status}] /review
├─ [{status}] /qa (条件: has_ui)
├─ [{status}] /ship
├─ [{status}] /ce:compound (可选)
└─ [{status}] /document-release
```

### finish.md 新增段落

位置：ship 段落之后、document-release 段落之前。

流程：

1. **可用性检查**：确认 `/ce:compound` skill 可用
   - 不可用 → 显示 "ce:compound 不可用，跳过经验沉淀"，将 `"compound"` 加入 `skipped_steps`，跳到 document-release

2. **引导式提问**（判断是否有经验值得记录）：
   > "在回答之前回顾一下本次开发过程：
   > - 遇到了意外的坑或 debug 难题吗？
   > - 发现了什么新的最佳实践或更好的做法？
   > - 有什么下次想避免或改进的？"

3. **根据回答**：
   - 全部"否" → 将 `"compound"` 加入 `skipped_steps`，设 `current_step` 为 `"document-release"`
   - 任一"是" → 提示用户执行 `/ce:compound`

4. **用户完成后**：
   - 将 `"compound"` 加入 `completed_steps`
   - 设 `current_step` 为 `"document-release"`
   - 更新 state.json + 重写 handoff.md

### 向后兼容

v1.0 项目的 state.json 不会有 `compound` 在 `completed_steps` 或 `current_step` 中。如果 v1.0 项目的 `current_step` 已经是 `"document-release"`，映射不变，自然兼容。无需迁移逻辑。

## 改动 2：brainstorming 前置知识搜索

### 位置

execute.md 的 `## brainstorming` 段落开头，在"提示用户执行 `/brainstorm`"之前。

### 流程

1. **检查 `docs/solutions/` 是否存在**
   - 不存在 → 显示 "未找到历史经验库（docs/solutions/），从零开始 brainstorming"，跳到步骤 3

2. **存在 → 派发 `learnings-researcher` agent**：
   - 输入：从 `Update_Plan.md`（existing）或 `product-requirements.md`（new）中提取项目主题作为搜索上下文
   - agent prompt：`"搜索 docs/solutions/ 中与以下主题相关的历史经验：{项目主题摘要}。返回相关的 solution 文件路径和关键要点摘要。"`
   - 有结果 → 写入 `.gspowers/artifacts/prior-learnings.md`，在 state.json `artifacts` 中记录（key: `"prior-learnings"`），显示 "找到 {N} 条相关历史经验，已写入 prior-learnings.md"
   - 无结果 → 显示 "未找到相关历史经验，从零开始 brainstorming"，不生成 prior-learnings.md
   - agent 派发失败 → 静默降级，等同无结果

3. **提示用户执行 brainstorming**：
   - 有 `prior-learnings.md` 时：
     ```
     /brainstorm

     请读取以下文件作为输入：
     - .gspowers/artifacts/architecture.md（架构）
     - .gspowers/artifacts/prior-learnings.md（历史经验）
     ```
   - 无 `prior-learnings.md` 时：
     ```
     /brainstorm

     请读取 .gspowers/artifacts/architecture.md 作为架构输入。
     ```

### 设计决策

- **调用者是 gspowers 自身**（不是用户手动调用）：知识搜索是无副作用的只读预处理，属于导航员原则的合理扩展。这是导航员模式的唯一例外。
- **使用 `learnings-researcher` agent**（不是简单 grep）：CE 的 learnings-researcher 理解 `docs/solutions/` 的 YAML frontmatter 结构，搜索质量更高。
- **无结果时简短提示后跳过**：让用户知道机制存在，随着 `docs/solutions/` 积累逐渐感受到价值。

## 改动 3：init 流程简化

### 变更范围

仅影响 `project_type = "existing"` 的流程。新项目流程不变。

### init.md 变更

替换 existing 项目中"告诉用户手动创建 Update_Plan.md"的流程为：

1. **检查 `Update_Plan.md` 是否已存在**
   - 已存在 → 显示 "检测到已有 Update_Plan.md，使用现有文件继续"，跳到正常的 state.json 创建流程

2. **不存在 → 询问用户**：
   > "请用自然语言描述你这次要做什么更新。
   > 比如：修复什么问题、添加什么功能、优化什么流程。
   > 尽量说清楚问题现状、期望结果和约束条件。"

3. **生成 Update_Plan.md 内容**：根据用户描述填充 4 个字段（问题现状、期望结果、约束条件、影响范围）

4. **展示确认**：
   > "以下是根据你的描述生成的更新计划，请确认或修改：
   > [展示完整内容]
   > 确认无误？（是 / 需要修改）"

5. **用户确认 → 写入 `Update_Plan.md`**；用户要修改 → 根据反馈调整后重新展示，直到确认

6. 继续正常的 state.json 创建流程

### SKILL.md description 变更

在 description 中加入"可生成脚手架文件"澄清（满足 eng review 决定 #2）。
